Platform

A shared service for running AI agents. The key idea in five words: an agent is configuration, not code. The platform conducts — streaming, memory, tools, guardrails — so product teams only write the score.

II.

Configuration, not code

Andante — the one idea to remember

An agent is exactly three things, set up in an admin UI. No new service, no new repository. Once defined, the platform runs it — everything else is built once and shared by every agent.

What you defineit's a form, not a codebase

Instructions

The prompt — who the agent is, what it knows, how it behaves.

Model

Picked in configuration. Switching models is a settings change, not a code change.

Tools

The MCP servers and built-in tools the agent is allowed to use — nothing more.

For the developers in the room: production apps integrate from their own UI through the typed npm client — @infotrack/agent-orchestrator-client — or the invoke REST API. Conversation state is just a session id you hold on to.

III.

What every agent gets on day one

Vivace — the table stakes, free

Streaming chat

Responses render as they're generated — target under two seconds to the first word.

Session memory

Conversations pause, resume and archive. Long chats stay in-limit via a rolling window of recent messages plus a saved summary of the older ones.

Tool guardrails

Every MCP tool is classified read-only or write. Read-only tools run automatically; write tools wait for the user's approval in chat.

Built-in tools

Web search, planning, scratchpads, artifacts and email drafting — available to any agent, no setup.

File attachments

Documents and images the model reads as part of your message. Working memory — not a knowledge base.

Visible reasoning

Watch the agent think and see every tool call, stored with the session — so answers are explainable later.

IV.

The pause that matters

Fermata — hold until the human decides

An agent can look things up on its own — but the moment it wants to change something, the run genuinely ends and waits for a person. Watch it happen (this is a simulation of the real playground behaviour):

Note what happened at the pause: nothing was spinning. The stream closed with the approval request as the last part of the message. Approve or Deny resubmits the conversation and the run continues from exactly that point — and a denial isn't an error: the model receives the refusal and adapts. There is no timeout on the decision.

VI.

Safe, governed — and honest about scope

Grave — the slide that matters most

Two principles are designed in, not bolted on: humans stay in control — every write action is an explicit human decision — and costs stay predictable, with token budgets per user, per device and per conversation, metered centrally at the gateway. Under the hood: model output is sanitised before render, deleted sessions stay recoverable for 30 days, and an access-control matrix defines who can call what.

The platform is deliberately v1-scoped: request-driven, one agent per session, extensible through tools. Here's the honest boundary map:

Can it…TodayThe path right now
Stream on invokeYesstream: true gives SSE with step, text-delta and tool events.
Read imagesPartialImage input works in chat with vision models. No generation, no voice.
Search a knowledge baseVia MCPNo built-in RAG — retrieval is a tool. Stand up an MCP server over your index; read-only, so it auto-executes.
Run multi-agent handoffsNot yetOne agent per session. Your app can orchestrate several via invoke calls.
Run on a scheduleNot yetTrigger invoke from your own scheduler. The platform never self-initiates — also a governance feature.
Remember across sessionsNot yetPer-session memory only. Inject durable context via instructions or at invoke time.
Return JSON-schema outputNot yetPrompt for JSON and validate caller-side; retry on parse failure.
Version agentsNot yetAgents update in place. Keep prompt text in git; treat the admin UI as deployment.

When a gap question lands: that's exactly the kind of thing that should be built once in the platform rather than five times in products — bring the use case to the team as a roadmap item.

VII.

From idea to prototype inside a day

Presto — the short path

1

Bring up the stack

One command starts everything locally — the API, MongoDB, Redis — in about a minute.

2

Describe your agent

Prompt, model, tools — in the admin UI. It's a form, not a codebase.

3

Try it in the playground

Create an agent, give it tools, watch it think. The fastest way to build intuition about what agents could do for your product area.

4

Ship from your own UI

Install @infotrack/agent-orchestrator-client from npm — or call the REST API directly. The AgentPlatform folder has the docs, specs and ADRs, and the team will help you scope a first agent.
