The Agent Platform
for new developers

AgentPlatform is InfoTrack's agent orchestration hub — a chat/agent experience where each configured "Agent" has its own model, instructions, and toolset, exposed both to interactive users and to external callers via an invoke API. This guide gets a new dev from zero to being able to trace a request end-to-end and make a confident first change.

§1The one thing to get right immediately

AgentPlatform and MaestroAi's own native chat assistant are two different products that happen to share the same MaestroAi .NET backend repo. Mixing them up is the single easiest mistake to make here if you don't internalize this split first.

AgentPlatformMaestroAi's own chat (Opus)
FrontendReact SPA (separate AgentPlatformUI repo)Embedded in MaestroAi's own product surfaces
OrchestrationNode.js BFF, AgentPlatform/src/node-server (Hono + Vercel AI SDK)C#, MaestroAi.Infrastructure.Agents (AgentService, OpusAgentBuilder) + legacy SK (KernelRepository)
Domain modelMaestroAi.Domain.AgnosticAgent.Models (Agent, Session, Message, Artifact)MaestroAi.Domain.Chat.Entities (a different Session class)
PurposeGeneral-purpose configurable agent hubProperty/conveyancing ordering assistant (title searches, company extracts, milestones)
MaestroAi's rolePure persistence/data layer, called over HTTP by the Node BFFMaestroAi is the runtime — no separate BFF

Rule of thumb: if a file lives under MaestroAi.Domain/Infrastructure/Application/Api.Controllers.AgnosticAgent.*, it's AgentPlatform's data layer. If it lives under MaestroAi.Infrastructure.Agents (AgentService.cs, OpusAgentBuilder.cs) or references OpusContext/OpusChatContext, it's the other product. Same repo, same C# namespace root, genuinely different systems.

§3Request lifecycle — trace this first

The fastest way to actually understand the system is to read chat.ts top to bottom and follow every call it makes. In order:

  1. Client → BFF: React SPA POSTs { messages, sessionId } to POST /api/chat (routes/chat.ts).
  2. Auth resolution: getRequestAuthContext(c) (lib/auth-context.ts) determines the caller mode — authenticated (Bearer/cookie → resolved to a loginId), device (session-cookie device login), or anonymous (rejected).
  3. Load agent config: dataLayerClient.getSession(sessionId) then dataLayerClient.getAgent(session.agentId) — this is the HTTP round-trip to MaestroAi's AgentsController. If session.agentId is missing, the request fails closed with a 400 rather than silently using defaults.
  4. Build the system prompt: buildSystemPreamble() pushes agentConfig.instructions plus a list of any existing artifacts in the session.
  5. Gather tools: getAgentMcpServerNames(agentConfig) + getMcpToolsForServers() (agent/mcp.ts) filters the agent's configured MCP servers and wires them via @ai-sdk/mcp's StreamableHTTPClientTransport; getFilteredBuiltInTools() adds built-ins. Write-classified tools get needsApproval: true.
  6. Run the model: streamText({ model: activeModel, messages, tools, stopWhen: [stepCountIs(maxSteps), ...] }) — maxSteps comes from agentConfig.maxSteps (default 10), genuinely enforced here.
  7. Stream back: result.toUIMessageStreamResponse() — AI SDK's UI Message Stream protocol over SSE.
  8. Persist (fire-and-forget): onFinish persists the user/assistant messages, tool invocations, and any created/updated artifacts back to MaestroAi.

routes/invoke.ts is the external-facing sibling of this same flow (POST /api/agents/{agentId}/invoke) — same agent-config resolution and tool-gathering logic, but session auto-creation, no approval prompts (everything auto-executes), and a choice between a single JSON response or a hand-rolled named-SSE-event stream. It's the closest existing thing to an A2A message endpoint, which is exactly why it's the reuse target for the in-flight A2A work (§5).

§5What's actively being built: A2A (Agent2Agent protocol)

AgentPlatform is currently being extended to speak A2A, the open protocol for agent-to-agent interoperability (AgentCard discovery, JSON-RPC task lifecycle, streaming). Tracked under Jira epic AIOPS-3535. Current tickets, in build order:

  1. AIOPS-3616 — AgentCard discovery (GET /.well-known/agent.json), activating the dormant Agent.A2AConfig column
  2. AIOPS-3618 — agent-to-agent authentication (extends machine-auth-client.ts)
  3. AIOPS-3617 — task lifecycle & JSON-RPC message surface, built on invoke.ts
  4. AIOPS-3619 — outbound A2A client (call another agent as a tool, alongside agent/mcp.ts)
  5. AIOPS-3541 — browsable catalogue + user-controlled A2A connections (the UI layer on top)

A separate team (Global Platform / Gateway) is independently building a Gateway-side A2A relay (InfoTrack.Platform.ModelContextProtocol repo, PR #509) for externally-hosted agents — AgentPlatform's own A2A work should eventually call through that relay rather than direct-to-upstream once it's available.

§7Hard-won lessons (read before your first PR)

  • Don't assume a class named AgentService or Session is AgentPlatform's — check the namespace (§1). MaestroAi.Infrastructure.Agents.AgentService and MaestroAi.Domain.Chat.Entities.Session belong to the other product.
  • Per-agent config (model/instructions/maxSteps/tools) already works end-to-end in both chat.ts and invoke.ts — don't rebuild this; if something seems unconfigured, check the actual Agent record in the data layer before assuming the plumbing is broken.
  • Ticket titles use AgentPlatform: / AgentPlatform — A2A: prefixes, not "Workstream N" — the old numbering had gaps once you separate AgentPlatform work from MaestroAi's own Opus-assistant tickets in the same Jira epic.
  • All inter-module communication is HTTP-only (Node ↔ .NET ↔ React) — no direct imports across module boundaries, per the constitution doc.
